#### HashMap
Структурно состоит из массива bucket'ов. Каждый **bucket - это изначально двусвязный** список. Bucket при добавлении значения вычисляется как `hashCode(key) & (n - 1)`, n - число bucket'ов(по умолчанию 16) 

[[Контракт equals() и hashCode()]]

**Скорость операций:**
- put / get / delete выполняются за O(1), в худшем случае коллизий за O(log n)

**Коллизия** возникает при совпадении hashCode() ключей. ==Алгоритм разрешения коллизии==:
1. С помощью `hashCode()` и `equals()` проводится проверка ключей. Если ключи равны, то значение bucket **перезаписывается**
2. Если ключи не равны, то старое и новое значения **связываются как ноды двусвязного списка**
3. После того, как количество элементов в хэше достигает определенного порога (8 элементов в одном bucket)происходит **переход к сбалансированным (красно-черным) деревьям** [[Tree]]. Что улучшает производительность в худшем случае с O(n) до O(log n).
4. hashCode(null) = 0

#### Точные числа treeify (важная деталь, часто путают)
- `TREEIFY_THRESHOLD = 8` — список → дерево при достижении 8 элементов **в одном bucket** (не 8 bucket'ов в таблице!)
- `UNTREEIFY_THRESHOLD = 6` — обратно дерево → список при **уменьшении до ≤6** элементов (при resize). Числа 8 и 6 разные специально — **гистерезис**, чтобы не было постоянных переключений список↔дерево при колебании количества элементов около границы
- `MIN_TREEIFY_CAPACITY = 64` — treeify произойдёт только если общий размер таблицы (`table.length`) ≥ 64. Если меньше — вместо treeify делается **resize всей таблицы** (это дешевле и сразу уменьшает коллизии во всех bucket'ах, а не в одном)

> Почему именно 8: при разумном hashCode() длина bucket распределена по Пуассону — вероятность bucket длиной 8 уже исчезающе мала, до этого порога дерево не оправдывает оверхед на память/перебалансировку

#### Как считается индекс bucket
- Индекс = `(n - 1) & hash`, где `n` — размер таблицы (table.length, **всегда степень двойки**)
- Перед этим исходный `hashCode()` **спредится**: `hash = h ^ (h >>> 16)` — старшие 16 бит смешиваются с младшими через XOR, чтобы при маленьком `n` (когда используются только младшие биты) различия в старших битах hashCode всё равно влияли на индекс и не создавали лишних коллизий
- `(n - 1) & hash` даёт тот же результат, что `hash % n`, **только когда `n` — степень двойки** (тогда `n-1` — это маска из всех единиц в младших битах) — поэтому размер таблицы HashMap всегда степень двойки, а не произвольное число

**Получение значения** по ключу в hashMap
1. По `hashCode(key)` вычисляется bucket, содержащий значение
2. Итерация по двусвязному списку пока key по `equals()` не совпадет с искомым ключом или пока указатель на следующий элемент не равен null

#### LinkedHashMap
Состоит из **двусвязного списка bucket'ов**. Сложности и алгоритм разрешения коллизий аналогичен HashMap
#### TreeMap
Состоит из сбалансированного красно-черного дерева [[Tree]].
Может хранить null только если используется comparator, разрешающий null

**Скорость операций:**
- put / get / delete выполняются за O(log n) за счет обхода красно-черного дерева
