#### Lazy vs Eager
- **Lazy** — связь подгружается только при первом реальном обращении к ней
- **Eager** — связь подгружается сразу вместе с родителем

> Важная деталь: до инициализации lazy-поле — это **НЕ `null`**, а **proxy-объект** (Hibernate подставляет туда заглушку, которая выглядит как реальный объект и подгружает данные через отдельный запрос при первом обращении к не-id методу)

#### `LazyInitializationException`
Возникает, когда обращение к lazy-полю происходит **после закрытия persistence context** (Hibernate `Session`).

**Типичный сценарий:**
```
1. Открыли сессию на REST-запрос (началась транзакция)
2. Получили entity, lazy-поля — proxy-заглушки
3. Транзакционный метод завершился → сессия закрылась
4. Контроллер/сериализатор обращается к lazy-полю
5. Proxy пытается дозагрузить данные -> сессии уже нет -> LazyInitializationException
```
Прокси умеет дозагрузить данные только пока активна сессия — после её закрытия запрос выполнить уже нельзя.

#### 3 правильных способа избежать (без OSIV и без огульного EAGER)
1. **`JOIN FETCH`** (JPQL) или **`@EntityGraph`** — заранее подгрузить нужные данные одним запросом, пока сессия открыта
   ```java
   @Query("SELECT o FROM Order o JOIN FETCH o.items WHERE o.id = :id")
   Optional<Order> findByIdWithItems(@Param("id") Long id);
   ```
2. **DTO-проекция** — замапить нужные поля в DTO внутри транзакционного метода (пока сессия открыта), возвращать DTO, а не entity
3. **`Hibernate.initialize(entity.getItems())`** — явно инициализировать нужную lazy-связь внутри транзакционного метода перед возвратом

> Почему `OSIV` (Open Session In View) и сплошной `EAGER` — анти-паттерны, а не решение: OSIV держит сессию открытой до рендеринга вида, расширяя транзакцию/коннекшен на весь HTTP-запрос — маскирует N+1 в слое представления, плохо под нагрузкой. EAGER-везде — лишние join/запросы там, где данные не нужны, нет гибкости на уровне конкретного use case, риск `MultipleBagFetchException` при нескольких `@OneToMany` коллекциях

#### См. также: проблема N+1
Загрузка N родительских сущностей с lazy-связью → при обращении к связи у каждой — по 1 доп. запросу (1 + N запросов вместо одного). Обнаружить — включить SQL-логирование (`logging.level.org.hibernate.SQL=debug`) и посчитать повторяющиеся запросы. Лечится тем же `JOIN FETCH`/`@EntityGraph`, либо `@BatchSize`/`hibernate.default_batch_fetch_size` (батчит lazy-подгрузку через `WHERE id IN (...)`).
