#### `volatile`
Даёт **две гарантии**:
1. **Видимость** — запись в volatile-переменную одним потоком немедленно видна другим потокам при чтении (не зависает в локальном кэше/регистрах потока неопределённо долго)
2. **Запрет переупорядочивания** — компилятор/JIT/CPU не могут переставить инструкции вокруг записи/чтения volatile (барьеры памяти)

**Не даёт атомарности составных операций.** `volatile int i; i++;` НЕ атомарен — инкремент это 3 шага (прочитать, увеличить, записать), между которыми другой поток может вклиниться. `volatile` тут не помогает.

> Частая ловушка: «добавим volatile» к счётчику не решает гонку на `count++`. `volatile` решает только видимость, не атомарность. Для атомарного инкремента нужен `AtomicInteger`/`synchronized` (см. ниже)

**happens-before:** если поток A пишет volatile-переменную, а поток B потом читает её — всё, что поток A изменил **до** записи (включая обычные не-volatile переменные), становится видимым потоку B **после** чтения. Это и есть гарантия порядка #2 на практике.

**volatile одного НЕ хватает** для double-checked locking (ленивая инициализация singleton) без правильной структуры:
```java
public class Singleton {
    private static volatile Singleton instance;   // volatile обязателен!
    public static Singleton getInstance() {
        if (instance == null) {                    // 1-я проверка без лока
            synchronized (Singleton.class) {
                if (instance == null) {            // 2-я проверка под локом
                    instance = new Singleton();
                }
            }
        }
        return instance;
    }
}
```
Без `volatile` на поле возможна публикация **частично сконструированного** объекта (другой поток увидит ссылку до того, как конструктор полностью отработал из-за переупорядочивания). Альтернатива DCL — **holder idiom** (статический вложенный класс, инициализация которого гарантированно ленивая и потокобезопасная за счёт самой механики загрузки классов JVM).

#### `synchronized` vs `ReentrantLock`
`ReentrantLock` (`java.util.concurrent.locks`) умеет сверх `synchronized`:
- `tryLock()` / `tryLock(timeout)` — неблокирующая или с таймаутом попытка захвата (можно не ждать вечно)
- `lockInterruptibly()` — захват, прерываемый по `interrupt()` (ожидание монитора `synchronized` прервать нельзя)
- **fairness** — конструктор `new ReentrantLock(true)` даёт FIFO-порядок выдачи лока, предотвращает starvation (дороже по производительности)
- несколько `Condition` на одном Lock через `newCondition()` — аналог `wait/notify`, но можно иметь несколько независимых очередей ожидания вместо одной на монитор

**Минусы ReentrantLock:** нужно явно `lock()`/`unlock()` в `finally` — забыть `unlock()` это баг (synchronized освобождает монитор автоматически даже при исключении). В современных JVM производительность сопоставима (biased/lightweight locking оптимизации) — выбор обычно по фичам, не по скорости.

Оба механизма **reentrant** — поток, уже владеющий локом, может захватить его повторно без deadlock с самим собой (счётчик удержаний инкрементируется).

`ReadWriteLock` — отдельный вид Lock, когда много читателей и редкие писатели (читатели не блокируют друг друга).

#### `ThreadPoolExecutor`
Параметры: `corePoolSize`, `maximumPoolSize`, `workQueue`, `keepAliveTime`, `RejectedExecutionHandler`.

**Порядок срабатывания при новой задаче:**
1. Активных потоков < `core` → создать новый поток (даже если есть свободные)
2. `core` занят → положить задачу в очередь (`workQueue`)
3. Очередь полна И потоков < `max` → создать дополнительный поток (вплоть до `max`)
4. Очередь полна И потоков = `max` → вызвать `RejectedExecutionHandler` (по умолчанию `AbortPolicy` — бросает `RejectedExecutionException`)

Другие стратегии `RejectedExecutionHandler`: `CallerRunsPolicy` (выполнить в вызывающем потоке — естественный backpressure), `DiscardPolicy`, `DiscardOldestPolicy`.

> `Executors.newFixedThreadPool(n)` использует `LinkedBlockingQueue` **без ограничения размера**. Из-за этого шаг 3 (создание потоков сверх core) фактически никогда не наступает — очередь "поглощает" любую нагрузку и растёт неограниченно → риск OOM под нагрузкой, который не виден до продакшена. Правильная настройка: `ArrayBlockingQueue` (bounded) + разумный `max` + `CallerRunsPolicy`/мониторинг отказов вместо бесконечного накопления

#### `AtomicInteger` и CAS
`AtomicInteger.incrementAndGet()` атомарен в отличие от `count++`, потому что использует **CAS (Compare-And-Swap)** — атомарную инструкцию CPU: сравнить текущее значение с ожидаемым и заменить новым **одной неделимой операцией**. Если другой поток успел изменить значение между чтением и попыткой записи — CAS не срабатывает, и операция **повторяется в цикле** (lock-free, в отличие от блокировки).

**Пример типичной ошибки (mini-review):**
```java
public class HitCounter {
    private int count = 0;
    public void hit() { count++; }          // НЕ атомарно — race condition
    public int getCount() { return count; }
}
```
- Дефект: `count++` — read-modify-write из 3 шагов, при конкурентных вызовах `hit()` возможна потеря инкрементов (lost update)
- `volatile int count` **не решает** проблему — даёт видимость, но не атомарность инкремента
- Правильный фикс: `AtomicInteger count = new AtomicInteger(); count.incrementAndGet();` (либо `synchronized` на обоих методах, но это медленнее)
