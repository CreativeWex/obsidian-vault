**Идемпотентность** — повторный одинаковый запрос (ретрай из-за таймаута, двойной клик) не должен приводить к повторному выполнению бизнес-эффекта (например, повторному списанию денег).

> Частая ошибка: проверять дубликат по самим данным запроса (сумма, счёт и т.п.). Это не работает — два **разных** легитимных платежа с одинаковыми данными (например, два перевода другу на одну и ту же сумму в течение дня) неотличимы от повторной отправки одного и того же запроса. Нужен отдельный идентификатор **намерения**, не привязанный к бизнес-данным

#### Дизайн idempotency key
1. **Клиент** генерирует уникальный key (UUID) **до** первого запроса и передаёт его в заголовке (`Idempotency-Key`) при каждой попытке, включая ретраи
	- Важно: генерирует именно **клиент**, а не сервер — если бы сервер генерировал key на каждый HTTP-запрос, ретрай получил бы новый key, и дедупликация не сработала бы
2. Сервер хранит таблицу `(idempotency_key UNIQUE, request_hash, status, response_body, created_at)`
3. **Алгоритм обработки:**
	- Попытка `INSERT` строки с этим key. **UNIQUE constraint** в БД гарантирует, что при гонке двух одновременных одинаковых запросов только один insert пройдёт
	- Insert успешен → выполнить бизнес-логику, сохранить результат, вернуть ответ
	- Key уже существует → **не повторять бизнес-логику**, вернуть **сохранённый ранее ответ** (тот же статус-код и тело) — это и есть идемпотентность по результату
4. Хранить key ограниченное время (TTL, например 24ч), затем можно очищать

#### Edge cases
- **Тот же key, другое тело запроса** — явная ошибка использования ключа клиентом. Сравнить `request_hash` текущего тела с сохранённым → вернуть 409/422
- **Гонка: второй запрос с тем же key пришёл, пока первый ЕЩЁ обрабатывается** (бизнес-логика не завершена) → вернуть 409 Conflict / 425 Too Early, либо подождать завершения первого — нельзя просто проигнорировать
- **Внешний побочный эффект** (например, вызов платёжного gateway) — одной транзакции в своей БД недостаточно для атомарности между "записать факт обработки" и "вызвать внешний API". Решение — **Outbox pattern**: сначала транзакционно зарезервировать intent в своей БД, потом вызвать внешний API, потом финализировать статус
